# Coding Task Precheck (必读)

> 每次执行改代码任务前，先完整阅读本文件并按清单逐项确认。

## 本次重构教训

1. 先做行为等价迁移，再做优化。
- 先严格对齐现有实现（接口、日志、命名、发布行为），确保可运行、可回归。
- 未经确认，不引入“顺手优化”“风格重构”“额外抽象”。

2. 外部契约禁止擅改。
- 发布包名、目录结构、启动命令、接口路径、返回字段、日志格式都属于对外契约。
- 任何对外可见改动必须先征求确认。

3. 修问题必须最小改动。
- 只改问题相关代码，不扩散、不叠加复杂策略。
- 能用一处小修复解决，就不要引入新层级/新文件/新机制。

4. 命名和注释以读者为中心。
- 函数名直接描述行为，不用作者视角命名。
- 注释只写职责和关键意图，不写空话，不写误导性背景。

5. 模块化适度，不为拆而拆。
- 只有当边界清晰且降低理解成本时才拆分。
- 小项目优先直白、可读、可追踪。

6. 先确认需求边界再动手。
- 对“是否允许改发布流程/命名/风格”先确认。
- 不明确时，默认保持现状，不做主观扩展。

7. 禁止无意义的防御性编程。
- 不为“看起来更安全”而随手加兜底、回退、类型分支、默认值覆盖。
- 只有在复现实例明确、且用户同意时才增加防御性逻辑。

8. 命名必须精简，禁止堆词。
- 函数、接口、变量名优先短而明确，不写冗长短语。
- 不为“自解释”而拼接多余单词，避免臃肿命名。

9. 属性读取次数少时，禁止封装单行 getter。
- 若只是 `return obj.xxx` 且调用很少，直接读取属性，不额外包一层方法。
- 只有在复用明显、边界明确时，才允许提炼 getter/访问器。

## 执行前清单（每次任务先过一遍）

- [ ] 我是否先复述了“本次只改什么，不改什么”？
- [ ] 我是否识别了所有对外契约，并承诺不擅改？
- [ ] 我的方案是否是最小改动路径？
- [ ] 是否存在我主观添加的新机制/新抽象？如果有，是否已获确认？
- [ ] 我是否添加了无意义防御性措施（兜底/回退/过度校验）？如果有，是否已删除或获确认？
- [ ] 我这次新增命名是否存在堆词和冗长？如果有，是否已精简？
- [ ] 我是否明确了验收标准（功能、日志、构建、发布）？

## 迁移完成必检（每次迁移后必须执行）

- [ ] 全局搜索旧接口/旧函数名残留。
- [ ] 删掉已废弃实现与未使用文件。
- [ ] 前后端调用链路全对齐（只留一套路径）。
- [ ] 语法检查通过后再提交结果。

## 执行纪律

- 未经用户明确同意，不修改发布包名、发布产物命名规则、发布流程对外行为。
- 未经用户明确同意，不引入与当前问题无关的结构性重构。
- 未经用户明确同意，不添加无意义防御性编程（兜底、回退、过度校验）。
- 命名默认保持精简，不堆叠无意义单词。
- 对象属性读取场景，默认直接访问；禁止为少量调用新增单行 `return` getter。
- 每次提交说明必须包含：
  - 改了什么
  - 为什么这样改
  - 哪些内容刻意没改（保持兼容）
